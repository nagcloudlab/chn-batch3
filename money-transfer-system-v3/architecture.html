v2 vs v3 — Who is the Assembler?

v2 — main() is the Assembler

// YOU create and wire everything
AccountRepository repo =   AccountRepositoryFactory.getAccountRepository("jdbc"); Notification notif = new EmailNotification(); TransferService svc =   new TransferServiceImpl(repo, notif);
// Then use it
svc.transfer(100.0, "ACC123", "ACC456");

v3 — Spring is the Assembler

// Spring creates and wires everything
ConfigurableApplicationContext ctx =   new ClassPathXmlApplicationContext("mts-config.xml");

// Just ask for the bean and use it
TransferService svc =   ctx.getBean("transferService", TransferService.class);
svc.transfer(100.0, "ACC123", "ACC456");

// Spring handles shutdown too
ctx.close();

Spring XML Configuration — mts-config.xml

<!-- Infrastructure bean: connection pool config -->
<bean id="hikariConfig" class="com.zaxxer.hikari.HikariConfig">
    <property name="jdbcUrl" value="jdbc:postgresql://localhost:5432/mydatabase"/>
    ...
</bean>

<!-- Infrastructure bean: DataSource (connection pool) -->
<bean id="dataSource" class="com.zaxxer.hikari.HikariDataSource" destroy-method="close">
    <constructor-arg ref="hikariConfig"/> <!-- DI: inject config -->
</bean>

<!-- Application bean: Repository -->
<bean id="jdbcAccountRepository" class="...JdbcAccountRepository">
    <constructor-arg ref="dataSource"/> <!-- DI: inject datasource -->
</bean>

<!-- Application bean: Notification -->
<bean id="emailNotificationService" class="...EmailNotificationService"/>

<!-- Application bean: Service (wired with repo + notification) -->
<bean id="transferService" class="...TransferServiceImpl">
    <constructor-arg ref="jdbcAccountRepository"/> <!-- DI: inject repo -->
    <constructor-arg ref="emailNotificationService"/> <!-- DI: inject notif -->
</bean>

Key Spring Concepts Introduced in v3

IoC Container

ApplicationContext is the container. It takes over creating and wiring objects. Your code just declares what it needs.

Bean

An object managed by Spring. Defined in XML with <bean>. Spring controls its creation, wiring, and destruction.

Dependency Injection

<constructor-arg ref="..."> tells Spring to inject one bean into another. No new in your code.

Configuration Metadata

mts-config.xml is the recipe. It tells Spring what beans to create and how to wire them.

Singleton Scope

By default, Spring creates one instance per bean. All requests for transferService return the same object.

Lifecycle Management

destroy-method="close" ensures HikariDataSource connection pool is properly shut down.

Inversion of Control (IoC)

In v2, your code controlled object creation and wiring.
In v3, Spring controls it — you just describe what you need in XML.
