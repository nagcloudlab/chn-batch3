The Problem — JDBC Limitations

Why raw JDBC is not enough for enterprise applications

Too Much Boilerplate

Open connection, create statement, set parameters, execute, iterate ResultSet, close everything. Repeated for every query.

No ORM Support

Manual mapping between ResultSet columns and Java object fields. No automatic type conversion or relationship mapping.

Manual Connections

Connection management, pooling, and lifecycle must be handled manually. Risk of resource leaks.

No Lazy Loading

Related data must be fetched eagerly with JOINs or with additional queries. No on-demand loading.

No DB Independence

SQL is database-specific. Switching from MySQL to PostgreSQL requires rewriting queries.

Solution

ORM frameworks + JPA specification give us object-relational mapping, DB independence, and zero boilerplate.

Customer Entity — Rich Annotations

Showcasing the full variety of JPA mapping annotations

@Entity @Table(name = "customers") public class Customer { @Id private Long id; @Column(name = "customer_name") private String name; @Enumerated(EnumType.STRING) private CustomerGender gender; // MALE, FEMALE, OTHER @Temporal(TemporalType.DATE) private LocalDate dob; @ElementCollection(fetch = FetchType.EAGER) @CollectionTable(name = "customer_addresses") private List<Address> address; @OneToMany(mappedBy = "customer") private List<Account> accounts; }

@Embeddable — Reusable Value Objects

Not a separate entity — fields are embedded directly into the parent table

@Embeddable public class Address { @Column(name = "customer_street") private String street; @Column(name = "customer_city") private String city; @Column(name = "customer_state") private String state; @Column(name = "customer_zip") private String zip; }

Customer Side — @OneToMany

@OneToMany(mappedBy = "customer") private List<Account> accounts;

Inverse side — does NOT own the FK. Just references the owning field name.

Account Side — @ManyToOne

@ManyToOne(fetch = FetchType.LAZY) @JoinColumn(name = "customer_id") private Customer customer;

Owning side — HAS the foreign key column in the accounts table.

Rule: The @ManyToOne side always OWNS the relationship (it has the FK column)

FetchType — LAZY vs EAGER

When to load related entities

EAGER Loading

Loads everything immediately with the parent
N+1 problem risk — extra queries for each relationship
Default for @ManyToOne and @OneToOne
// Loads Customer immediately with Account @ManyToOne // default is EAGER private Customer customer;

LAZY Loading

Loads on first access only (proxy object)
Better performance — no unnecessary queries
Default for @OneToMany and @ManyToMany
// We override default to LAZY @ManyToOne(fetch = FetchType.LAZY) private Customer customer;
In our project: Account.customer uses FetchType.LAZY — we override the @ManyToOne default (EAGER) for better performance

Custom Queries — @Query & Method Names

Two powerful ways to define custom queries

@Query with JPQL

public interface AccountRepository extends JpaRepository<Account, String> { @Query("SELECT a FROM Account a WHERE a.customer.id = :customerId") List<Account> findByCustomer( @Param("customerId") Long customerId ); }
JPQL uses entity names (Account, Customer) not table names (accounts, customers)

Query Derivation from Method Names

Spring parses the method name and generates the query automatically!

findByName(String name)
findByBalanceGreaterThan(double bal)
findByCustomerIdAndBalance(Long, double)
findByHolderNameContaining(String s)
How? Spring reads findBy + PropertyName + Keyword and builds the WHERE clause

Service Layer — AccountService

Clean, simple service with constructor injection and @Transactional

@Service public class AccountService { private final AccountRepository accountRepository; // Constructor Injection — recommended approach public AccountService(AccountRepository accountRepository) { this.accountRepository = accountRepository; } @Transactional(readOnly = true) public List<Account> findAll() { return accountRepository.findAll(); } @Transactional(readOnly = true) public List<Account> findByCustomer(Long customerId) { return accountRepository.findByCustomer(customerId); } }
Constructor Injection
No @Autowired needed — Spring auto-injects single constructor
@Transactional
Manages transaction boundaries — commit on success, rollback on exception
readOnly = true
Optimization hint — Hibernate skips dirty checking for read operations
