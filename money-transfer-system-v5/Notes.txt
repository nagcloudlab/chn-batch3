v1 Problems
============

1. Design issue — tight coupling
   - TransferServiceImpl is tightly coupled to JdbcAccountRepository
   - Why it hurts:
     => unit-testing is difficult => bug-fix slow, dev-slow
     => can't extend with new features easily (e.g. swap JDBC with JPA)

2. Performance issue — wasteful object creation
   - Every transfer() call creates a new JdbcAccountRepository instance
   - Why it hurts:
     => resource usage high (memory, connections)
     => latency high => not responsive to user


Root cause
==========
   => Dependent creates its own dependency ( "new" inside the class )


v2 Solutions (step by step)
============================

Step 1: Introduce interface (AccountRepository)
   - Code depends on abstraction, not concrete class
   - Enables multiple implementations (Jdbc, Jpa, etc.)
   - Follows OCP — open for extension, closed for modification

Step 2: Use Factory to create (AccountRepositoryFactory)
   - Don't create dependency in dependent's home
   - Get it from a factory instead
   - Solves: tight coupling to a specific implementation

Step 3: Use Dependency Injection (Constructor Injection)
   - Don't create AND don't lookup from factory directly
   - Let a third-party inject the dependency
   - This is Inversion of Control (IoC) — the dependent gives up control of creating/finding its dependency

   Types of DI:
     - Constructor injection => for required dependencies
     - Setter injection     => for optional dependencies


OOP Concepts
=============
   - Abstraction    : what to do?    ( interface )
   - Encapsulation  : how to do?     ( implementation hidden )
   - Inheritance    : reuse code     ( is-a relationship )
   - Polymorphism   : one name, many forms ( overloading / overriding )


SOLID Principles
=================
   S — Single Responsibility  : a class should have only one reason to change
   O — Open-Closed            : open for extension, closed for modification
   L — Liskov Substitution    : subtypes must be substitutable for their base types
   I — Interface Segregation  : clients should not depend on interfaces they don't use
   D — Dependency Inversion   : high-level modules should not depend on low-level modules;
                                both should depend on abstractions

   => v2 applies: O, L, D

spring framework core concepts

- DI ( wiring dependencies )
------------------------------------------------

1. @Bean

2. @Component
   - @Repository => specialization of @Component for persistence layer
     - JdbcAccountRepository
   - @Service => specialization of @Component for service layer
     - TransferServiceImpl
   - @Controller => specialization of @Component for presentation layer
     - TransferController
   - @Configuration => specialization of @Component for configuration classes
     - MtsConfiguration

3. @Autowired 
   - constructor injection (recommended) 
   - setter injection
   - field injection (not recommended)

   Note: @Autowired no need to be used on constructor if there is only one constructor. 
   Spring will automatically use it for injection.

4. @Primary
   - when multiple beans of the same type exist, use @Primary to specify which one to inject by default
   - e.g. @Primary on JdbcAccountRepository to make it the default AccountRepository bean

5. @Qualifier
   - when multiple beans of the same type exist, use @Qualifier to specify which one to inject
   - e.g. @Qualifier("jdbcAccountRepository") to specify the JdbcAccountRepository bean

6. bean/component scope - @Scope
   - singleton (default) => one instance per Spring container
   - prototype => new instance every time requested
   - request => one instance per HTTP request (web applications)
   - session => one instance per HTTP session (web applications)   

7. @Lazy
   - delay bean initialization until it's needed
   - can improve startup time and resource usage if the bean is expensive to create and not always needed

8. @PostConstruct
   - method annotated with @PostConstruct will be called after the bean is initialized and dependencies are injected
   - useful for any initialization logic that needs to run after the bean is fully set up   

9. @PreDestroy
   - method annotated with @PreDestroy will be called before the bean is destroyed by the container
   - useful for any cleanup logic that needs to run before the bean is removed from the container

10. @Value
   - inject values from properties files or environment variables into fields or method parameters
   - e.g. @Value("${db.url}") to inject the value of "db.url" from the properties file

11. @PropertySource
   - specify the location of properties files to load into the Spring environment
   - e.g. @PropertySource("classpath:application.properties") to load properties from the classpath


12. @Profile
   - specify that a bean should only be created in certain profiles (e.g. dev, prod)
   - e.g. @Profile("dev") to only create the bean when the "dev" profile is active
   - activate profiles: applicationContext.getEnvironment().setActiveProfiles("dev")
   - use case: JdbcAccountRepository for "dev", JpaAccountRepository for "prod"

13. @Conditional
   - create beans only when a custom condition is met at runtime
   - implement the Condition interface and override matches() method
   - e.g. @Conditional(PostgresDriverCondition.class) — only create DataSource if PostgreSQL driver is on classpath
   - more flexible than @Profile — you define arbitrary logic

14. @EventListener
   - method annotated with @EventListener will be called when a matching ApplicationEvent is published
   - enables loose coupling — publisher doesn't know about listeners
   - e.g. EmailNotificationService listens for TransferCompletedEvent
   - to publish: applicationEventPublisher.publishEvent(new TransferCompletedEvent(...))
   - custom events extend ApplicationEvent

15. @Value with SpEL (Spring Expression Language)
   - beyond simple property injection, @Value supports SpEL expressions using #{...}
   - e.g. @Value("#{T(java.lang.Runtime).getRuntime().availableProcessors()}") => call static methods
   - e.g. @Value("#{systemProperties['user.home']}") => read system properties
   - e.g. @Value("#{mtsConfig.dbUrl}") => read a property from another bean
   - syntax: ${...} for property placeholders, #{...} for SpEL expressions

spring framework core concepts

- AOP ( Aspect-Oriented Programming )
------------------------------------------------

The Problem: Cross-Cutting Concerns
=====================================
   - Some logic cuts across multiple layers: logging, authorization, transactions, caching, metrics
   - Without AOP, this logic gets scattered and duplicated across business code
   - Violates SRP — business methods do business logic AND cross-cutting logic
   - Hard to maintain — change in one concern requires changes everywhere


Proxy Pattern (manual approach — ProxyPatternExample.java)
============================================================
   - TrainerProxy wraps the real Trainer object
   - Before calling the real method, the proxy runs authorization + logging
   - This is the essence of AOP — but done manually
   - Spring AOP automates this using dynamic proxies


AOP Terminology
================
   - Aspect         : a class that contains cross-cutting logic (e.g. AuthorizationAspect)
   - Advice         : what to do — the actual logic in the aspect method
   - Join Point     : where it can apply — any method execution in your code
   - Pointcut       : where it should apply — an expression that selects specific join points
   - Target         : the actual object being proxied (e.g. TransferServiceImpl)
   - Proxy          : the wrapper Spring creates around the target
   - Weaving        : the process of linking aspects to target objects (done at runtime by Spring)


16. @Aspect
   - marks a class as an aspect containing cross-cutting concern logic
   - must also be a Spring bean (@Component)

17. @EnableAspectJAutoProxy
   - placed on @Configuration class to enable Spring AOP proxy creation
   - without this, @Aspect classes are just regular beans — no proxying happens

18. Advice Annotations
   - @Before("pointcut")           => runs before the method
   - @AfterReturning("pointcut")   => runs after method returns successfully
   - @AfterThrowing("pointcut")    => runs after method throws an exception
   - @After("pointcut")            => runs after method (finally — always runs)
   - @Around("pointcut")           => wraps the method — runs before AND after
     - receives ProceedingJoinPoint to call joinPoint.proceed()
     - most powerful — can control whether method executes, modify return value, handle exceptions

19. Pointcut Expressions
   - execution(modifiers? return-type declaring-type? method-name(param-pattern) throws?)
   - e.g. execution(void com.npci.service.TransferService.*(..))
     => any void method in TransferService interface, any args
   - e.g. execution(void transfer(..))
     => any void method named "transfer", any args, any class
   - wildcards: * (any single), .. (any args/subpackages)

20. @Order
   - controls the execution order of multiple aspects
   - lower number = higher priority (runs first for @Before, last for @After)
   - e.g. @Order(1) AuthorizationAspect runs before @Order(2) AtomicTransactionAspect


Our v4 AOP Implementation
===========================

   AuthorizationAspect (@Order 1)
   - @Before("execution(void com.npci.service.TransferService.*(..))")
   - Runs authorization check before any TransferService method

   AtomicTransactionAspect (@Order 2)
   - @Around("execution(void transfer(..))")
   - begin transaction → proceed() → commit on success / rollback on exception → end

   Execution flow:
     main() → Spring Proxy → AuthorizationAspect.@Before
                            → AtomicTransactionAspect.@Around (begin)
                              → TransferServiceImpl.transfer() [actual business logic]
                            → AtomicTransactionAspect.@Around (commit/rollback)

   How Spring AOP works:
   - Spring creates a proxy around TransferServiceImpl
   - When you call getBean("transferService"), you get the PROXY, not the real object
   - The proxy intercepts calls and runs aspect advice before/after forwarding to the target
   - Uses JDK dynamic proxy (interface-based) or CGLIB (class-based)

spring framework core concepts

- Spring JDBC & Transaction Management
------------------------------------------------

21. JdbcTemplate
   - Spring's JDBC abstraction — eliminates boilerplate (connection mgmt, exception handling, resource cleanup)
   - Inject via constructor: @Autowired JdbcTemplate jdbcTemplate
   - queryForObject() — single row query with RowMapper lambda
   - update() — INSERT, UPDATE, DELETE operations
   - Configured via @Bean in MtsConfiguration: new JdbcTemplate(dataSource)

22. @Transactional
   - declarative transaction management — replaces manual begin/commit/rollback
   - Spring wraps the method in a transaction using AOP (proxy-based)
   - rollbackFor = RuntimeException.class — specify which exceptions trigger rollback
   - transactionManager = "transactionManager" — specify which tx manager to use
   - if method completes normally → auto-commit
   - if method throws specified exception → auto-rollback

23. @EnableTransactionManagement
   - placed on @Configuration class to enable @Transactional processing
   - Spring creates proxies around @Transactional methods
   - works via AOP — same proxy mechanism as @Aspect

24. PlatformTransactionManager
   - Spring's transaction abstraction interface
   - DataSourceTransactionManager — implementation for JDBC-based transactions
   - Configured as @Bean: new DataSourceTransactionManager(dataSource)


Spring Context Lifecycle (detailed)
=====================================
   - init phase:
     1. Bean definition loading (from @ComponentScan, @Bean methods)
     2. BFPP (BeanFactoryPostProcessor) execution — can modify bean definitions
     3. For each bean definition:
        a. Create bean instance (constructor)
        b. DI (constructor/setter/field injection)
        c. BPP.postProcessBeforeInitialization()
        d. @PostConstruct method execution
        e. BPP.postProcessAfterInitialization() — AOP proxy creation happens here
     4. Application is ready

25. BeanFactoryPostProcessor (BFPP)
   - runs BEFORE any bean is created
   - can inspect and modify bean definitions (metadata, not instances)
   - use case: override property values, add new bean definitions
   - implement BeanFactoryPostProcessor interface

26. BeanPostProcessor (BPP)
   - runs AFTER each bean is created but wraps the initialization phase
   - postProcessBeforeInitialization() — before @PostConstruct
   - postProcessAfterInitialization() — after @PostConstruct (proxy creation for AOP happens here)
   - use case: custom proxies, validation, metrics wrapping


Spring Annotations Summary (covered so far)
=============================================
   DI & Wiring       : @Bean, @Component, @Autowired, @Primary, @Qualifier
   Stereotypes        : @Repository, @Service, @Controller, @Configuration
   Scope & Lifecycle  : @Scope, @Lazy, @PostConstruct, @PreDestroy
   Configuration      : @Value, @PropertySource, @Profile, @Conditional
   Events             : @EventListener, ApplicationEvent, ApplicationEventPublisher
   AOP                : @Aspect, @EnableAspectJAutoProxy, @Before, @After, @AfterReturning, @AfterThrowing, @Around, @Order
   Spring JDBC       : JdbcTemplate, @Transactional, @EnableTransactionManagement, PlatformTransactionManager
   Lifecycle Hooks   : BeanFactoryPostProcessor (BFPP), BeanPostProcessor (BPP)
