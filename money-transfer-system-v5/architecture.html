Overview
What's New in v5
Four key additions over v4

JdbcTemplate

Replaces raw JDBC (Connection, PreparedStatement, ResultSet) with Spring's clean abstraction.

queryForObject() and update() — no more try/catch/finally boilerplate for resource cleanup.

@Transactional

Declarative transaction management replaces the manual AtomicTransactionAspect from v4.

Normal return → auto-commit. Exception → auto-rollback. Uses AOP under the hood!

BPP & BFPP

Understand Spring's internal extension points.

BeanFactoryPostProcessor modifies bean definitions. BeanPostProcessor wraps bean initialization (AOP proxies created here).

JdbcTemplate — Bye Bye Boilerplate
JdbcAccountRepository: raw JDBC vs JdbcTemplate

Before (v4 — raw JDBC)

Connection conn = dataSource.getConnection();
try {
  PreparedStatement ps = conn.prepareStatement(
    "SELECT number, balance FROM accounts WHERE number = ?");
  ps.setString(1, accountNumber);
  ResultSet rs = ps.executeQuery();
  if (rs.next()) {
    account.setAccountNumber(rs.getString("number"));
    account.setBalance(rs.getDouble("balance"));
  }
} finally {
  conn.close(); // don't forget!
}

Manual connection mgmt, exception handling, resource cleanup

After (v5 — JdbcTemplate)

String sql = "SELECT number, balance FROM accounts WHERE number = ?";

return jdbcTemplate.queryForObject(sql,   new Object[]{accountNumber},   (rs, rowNum) -> {     Account account = new Account();     account.setAccountNumber(rs.getString("number"));     account.setBalance(rs.getDouble("balance"));     return account;   });

No Connection/PreparedStatement/ResultSet/finally — Spring handles it all

queryForObject() — single row update() — INSERT/UPDATE/DELETE RowMapper lambda — maps ResultSet to Object
Transaction Management
@Transactional — Declarative Tx Management
Replaces v4's manual AtomicTransactionAspect with a single annotation

v5 — @Transactional

@Transactional(   transactionManager = "transactionManager",   rollbackFor = RuntimeException.class ) public void transfer(double amount,
    String from, String to) {

  // ONLY business logic
  Account fromAcct = repo.loadAccount(from);
  Account toAcct = repo.loadAccount(to);
  fromAcct.debit(amount);
  toAcct.credit(amount);
  repo.updateAccount(fromAcct);
  repo.updateAccount(toAcct);
}

v4 — Manual @Around Aspect

@Component @Aspect @Order(2)
public class AtomicTransactionAspect {

  @Around("execution(void transfer(..))")
  public void aroundTransfer(ProceedingJoinPoint jp) {
    try {       beginTransaction();       jp.proceed();       commitTransaction();     } catch (Throwable ex) {       rollbackTransaction();     }   }
}

Normal return → auto-commit

Exception thrown → auto-rollback

Under the hood → uses AOP proxy (same mechanism!)

Extension Point
BeanFactoryPostProcessor (BFPP)
Runs BEFORE any bean is created — inspects and can modify bean definitions
// @Component ← commented out (demo only)
public class BFPP implements BeanFactoryPostProcessor {

  @Override   public void postProcessBeanFactory(       ConfigurableListableBeanFactory beanFactory) {
    logger.info("BFPP called. Bean count: {}",
        beanFactory.getBeanDefinitionCount());

    for (String name : beanFactory.getBeanDefinitionNames()) {
      logger.info("Bean name: {}", name);
    }
  }
}

When does it run?

After bean definitions are loaded, BEFORE any bean instance is created

What can it do?

Inspect & modify bean definitions (metadata) — override properties, add definitions

What it cannot do

Cannot work with bean instances — they don't exist yet at this stage

Extension Point
BeanPostProcessor (BPP)
Wraps each bean's initialization phase — AOP proxy creation happens here
// @Component ← commented out (demo only)
public class BPP implements BeanPostProcessor {

  @Override   public Object postProcessBeforeInitialization(Object bean, String name) {     logger.info("Before Initialization : {}", name);
    return bean;
  }

  @Override   public Object postProcessAfterInitialization(Object bean, String name) {     logger.info("After Initialization : {}", name);
    return bean; // can return a PROXY instead!
  }
}

postProcessBeforeInitialization

Runs BEFORE @PostConstruct. Can inspect or modify the bean.

postProcessAfterInitialization

Runs AFTER @PostConstruct. AOP proxy creation happens here!

Use cases

Custom proxies, validation, metrics wrapping, @Transactional proxy

Summary
v5 Concepts — Quick Reference
Everything new in v5

JdbcTemplate

Spring's JDBC abstraction. Eliminates boilerplate. queryForObject(), update().

@Transactional

Declarative tx management. Replaces manual @Around aspect. Auto commit/rollback.

@EnableTransactionManagement

On @Configuration class. Enables @Transactional proxy creation.

PlatformTransactionManager

Spring's tx abstraction. DataSourceTransactionManager for JDBC.

BeanFactoryPostProcessor

Runs before any bean created. Inspects/modifies bean definitions.

BeanPostProcessor

Wraps bean init. before/after @PostConstruct. AOP proxies created in "after".

v4 → v5 Key Shift

Manual AtomicTransactionAspect with @Around → replaced by Spring's built-in @Transactional.
Raw JDBC with Connection/PreparedStatement → replaced by JdbcTemplate.
Understanding the lifecycle (BFPP, BPP) explains how Spring makes all this work.
