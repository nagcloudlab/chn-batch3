The Problem
Cross-Cutting Concerns
Some logic cuts across every layer — it doesn't belong to one class

Without AOP (scattered code)

public void transfer(...) {
  checkAuthorization(); // security   beginTransaction(); // tx mgmt   log("transfer start"); // logging
  // actual business logic
  from.debit(amount);
  to.credit(amount);

  commitTransaction(); // tx mgmt   log("transfer done"); // logging }

Business logic buried under infrastructure code. Violates SRP.

With AOP (clean business code)

public void transfer(...) {

  // ONLY business logic
  from.debit(amount);   to.credit(amount);
}

// Auth, Tx, Logging handled
// by separate @Aspect classes
// Spring auto-weaves them in

Clean separation. Business code stays pure. Concerns are modular.

Proxy Pattern — The Idea Behind AOP
ProxyPatternExample.java — manual approach before Spring AOP

Target Object (Trainer)

class Trainer {
  // Join Points — methods AOP can intercept
  public void getSpringBootTraining() {     System.out.println("Spring Boot Training");   }   public void getSQLTraining() {     System.out.println("SQL Training");   } }

Proxy (wraps Target + Aspects)

class TrainerProxy {
  Trainer trainer = new Trainer();
  Authrization auth = new Authrization();
  Logging log = new Logging();

  public void getSpringBootTraining() {
    auth.doAuthrization(); // Aspect 1     log.doLogging(); // Aspect 2     trainer.getSpringBootTraining(); // Target   }
}

This works, but you must write proxy code manually for every class. Spring AOP automates this!

Vocabulary
AOP Terminology
The key terms you need to know

Aspect

A class containing cross-cutting logic.
@Aspect + @Component

e.g. AuthorizationAspect, AtomicTransactionAspect

Advice

The action taken by an aspect at a join point. The "what to do".

e.g. @Before, @After, @Around methods

Join Point

A point in execution where an aspect can be applied.

In Spring AOP: always a method execution

Pointcut

An expression that selects which join points to intercept.

e.g. execution(void transfer(..))

Target

The actual object being advised (proxied).

e.g. TransferServiceImpl instance

Weaving

Linking aspects with target objects to create proxies.

Spring does this at runtime (dynamic proxies)

Pointcut
Pointcut Expressions
Tell Spring which methods the advice should apply to
execution(modifiers? return-type declaring-type? method-name(params) throws?)

Our AuthorizationAspect pointcut:

@Before("execution(void com.npci.service.TransferService.*(..))")

Match: any void method in TransferService interface, any arguments

Our AtomicTransactionAspect pointcut:

@Around("execution(void transfer(..))")

Match: any void method named transfer, any class, any arguments

*
any single token
(type, method name)
..
any number of args
or subpackages
void
match return type
(use * for any)

AuthorizationAspect — @Before
Runs authorization check before every TransferService method
@Component @Aspect @Order(1) // runs FIRST public class AuthorizationAspect {

  @Before("execution(void com.npci.service.TransferService.*(..))")   public void checkAuthorization() {
    logger.info(">>> checking authorization for transfer.");
  }

}

@Component

Makes it a Spring bean (discovered by @ComponentScan)

@Aspect

Tells Spring this bean contains AOP advice

@Order(1)

Runs before other aspects (lower = first)

Our Implementation
AtomicTransactionAspect — @Around
Wraps transfer() with begin / commit / rollback / end
@Component @Aspect @Order(2) // runs AFTER AuthorizationAspect
public class AtomicTransactionAspect {

  @Around("execution(void transfer(..))")
  public void aroundTransfer(ProceedingJoinPoint joinPoint)
      throws Throwable {
    try {
      logger.info(">>> begin transaction.");       joinPoint.proceed(); // call actual transfer()       logger.info(">>> commit transaction.");     } catch (Throwable ex) {
      logger.error(">>> rollback transaction: {}", ex.getMessage());       throw ex; // rethrow to propagate     } finally {
      logger.info(">>> end transaction.");     }
  }
}

ProceedingJoinPoint.proceed() is the key — it calls the actual target method.
Without calling proceed(), the target method never executes.

Ordering
@Order — Aspect Execution Sequence
Lower number = higher priority. Controls the nesting order of aspects.
Actual execution order when transfer() is called:
1. AuthorizationAspect @Before — "checking authorization"
2. AtomicTransactionAspect @Around — "begin transaction"
3.     TransferServiceImpl.transfer() — actual business logic
4. AtomicTransactionAspect @Around — "commit transaction"
5. AtomicTransactionAspect @Around — "end transaction"

Separation of Concerns

In v3, cross-cutting logic (auth, tx) was inside business methods.
In v4, aspects handle it separately — business code stays clean and focused.

What v4 (AOP) Gains Over v3

Clean Business Code
transfer() only has business logic, nothing else
Modular Concerns
Auth, Tx, Logging each in their own @Aspect class
No Code Duplication
One aspect applies to all matching methods
Easy to Enable/Disable
Remove @Component or change @Order to reconfigure
Follows SRP
Each class has one reason to change
Foundation for Spring TX
