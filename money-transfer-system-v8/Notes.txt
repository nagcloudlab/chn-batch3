Spring Data JPA
================

What is Spring Data JPA?
--------------------------
   - Spring Data JPA builds on top of JPA (Jakarta Persistence API) and Hibernate
   - provides repository interfaces that auto-generate common CRUD queries
   - no need to write SQL or implementation classes for basic operations
   - just extend JpaRepository and Spring creates the implementation at runtime


JpaRepository
--------------
   - interface from Spring Data JPA
   - extends CrudRepository and PagingAndSortingRepository
   - provides: save(), findById(), findAll(), deleteById(), count(), existsById(), etc.
   - you only define the interface — Spring generates the implementation at runtime

   e.g.
     public interface AccountRepository extends JpaRepository<Account, String> {
     }

     => Account is the entity type
     => String is the type of the primary key (@Id field)
     => you get save(), findById(), findAll(), delete() etc. for free — no implementation class needed


v7 vs v8 — Repository comparison
-----------------------------------

   v7 (JDBC):
     @Repository
     public class JdbcAccountRepository implements AccountRepository {
         private final JdbcTemplate jdbcTemplate;
         // manual SQL queries with jdbcTemplate.query(), jdbcTemplate.update()
         // 50+ lines of boilerplate code
     }

   v8 (JPA):
     public interface AccountRepository extends JpaRepository<Account, String> {
     }
     => 1 line. Spring Data JPA generates the implementation.
     => no SQL, no JdbcTemplate, no RowMapper, no implementation class



JPA Entity Annotations
=======================

@Entity
--------
   - marks a class as a JPA entity (mapped to a database table)
   - required on every class that should be persisted to the database
   - from jakarta.persistence.Entity

@Table(name = "accounts", schema = "public")
-----------------------------------------------
   - specifies the database table name and schema
   - optional if class name matches table name
   - from jakarta.persistence.Table

@Id
----
   - marks the primary key field
   - every @Entity must have one @Id field
   - from jakarta.persistence.Id

@Column(name = "number", nullable = false, unique = true)
-----------------------------------------------------------
   - maps a field to a specific database column
   - optional if field name matches column name
   - can specify constraints: nullable, unique, length, etc.
   - from jakarta.persistence.Column

@GeneratedValue(strategy = GenerationType.IDENTITY)
------------------------------------------------------
   - auto-generates the primary key value
   - IDENTITY strategy => database auto-increment
   - used on Transaction.id — database generates the ID
   - from jakarta.persistence.GeneratedValue

@ManyToOne
-----------
   - defines a many-to-one relationship between entities
   - many Transactions belong to one Account
   - from jakarta.persistence.ManyToOne

@JoinColumn(name = "account_number", referencedColumnName = "number")
-----------------------------------------------------------------------
   - specifies the foreign key column
   - name = column in this table (transactions.account_number)
   - referencedColumnName = column in the referenced table (accounts.number)
   - from jakarta.persistence.JoinColumn

@Enumerated(EnumType.STRING)
------------------------------
   - maps an enum field to a database column
   - EnumType.STRING => stores the enum name as a string (e.g., "DEPOSIT", "WITHDRAWAL")
   - EnumType.ORDINAL => stores the enum ordinal (0, 1, ...) — avoid this, it's fragile
   - from jakarta.persistence.Enumerated

@Temporal(TemporalType.TIMESTAMP)
-----------------------------------
   - specifies the type for date/time fields
   - TIMESTAMP => includes date + time
   - DATE => date only
   - TIME => time only
   - from jakarta.persistence.Temporal


Account Entity (v8)
--------------------
   @NoArgsConstructor
   @AllArgsConstructor
   @Setter
   @Getter
   @Entity
   @Table(name = "accounts", schema = "public")
   public class Account {

       @Id
       @Column(name = "number", nullable = false, unique = true)
       private String accountNumber;

       @Column(name = "holder_name")
       private String accountHolderName;

       private double balance;
   }


Transaction Entity (new in v8)
-------------------------------
   @Entity
   @Table(name = "transactions", schema = "public")
   public class Transaction {

       @Id
       @GeneratedValue(strategy = GenerationType.IDENTITY)
       private Long id;

       private Double amount;

       @Temporal(TemporalType.TIMESTAMP)
       private LocalDateTime timestamp;

       @Column(name = "type")
       @Enumerated(EnumType.STRING)
       private TransactionType type;

       @ManyToOne
       @JoinColumn(name = "account_number", referencedColumnName = "number")
       private Account account;
   }

       DEPOSIT,
       WITHDRAWAL
   }



Custom JPQL Queries with @Query
=================================

   - JPQL = Jakarta Persistence Query Language (object-oriented, not SQL)
   - queries on entity classes and fields, not table names and columns
   - Spring Data JPA executes the query and maps results

   e.g.
     public interface TransactionRepository extends JpaRepository<Transaction, Long> {

         @Query("SELECT t FROM Transaction t WHERE t.account.accountNumber = :accountNumber ORDER BY t.timestamp DESC")
         List<Transaction> findByAccountNumber(String accountNumber);
     }

     => "Transaction t" refers to the Transaction entity, not the table name
     => "t.account.accountNumber" navigates the @ManyToOne relationship
     => ":accountNumber" is a named parameter bound from the method argument



@Transactional — Advanced Attributes
======================================

Full usage in TransferServiceImpl:

   @Transactional(
       transactionManager = "transactionManager",
       isolation = Isolation.READ_COMMITTED,
       rollbackFor = RuntimeException.class,
       noRollbackFor = AccountNotFoundException.class,
       timeout = 30,
       readOnly = false
   )

   Attributes explained:
   - transactionManager   => which transaction manager bean to use (default is "transactionManager")
   - isolation             => transaction isolation level
                             READ_COMMITTED => prevents dirty reads
                             Other levels: READ_UNCOMMITTED, REPEATABLE_READ, SERIALIZABLE
   - rollbackFor           => which exceptions trigger a rollback
                             RuntimeException.class => rollback on any runtime exception
   - noRollbackFor         => exceptions that should NOT trigger rollback
                             AccountNotFoundException.class => don't rollback if account not found
   - timeout               => max time (seconds) before transaction is automatically rolled back
   - readOnly              => hint to the database that this transaction is read-only (optimization)
                             false => this transaction writes data



Transaction Propagation
========================

   - defines how transactions behave when one @Transactional method calls another
   - Propagation.REQUIRED (default) => join existing transaction, or create new one if none exists

   e.g. in FooService:

     @Service
     public class FooService {

         @Transactional(propagation = Propagation.REQUIRED)
         public void doWriteWithDatabase() {
             // runs inside the SAME transaction as the caller (TransferServiceImpl.transfer())
             // if this fails, the entire transfer is rolled back
         }
     }

   TransferServiceImpl.transfer() [TX starts here]
       -> accountRepository.save(fromAccount)      [same TX]
       -> accountRepository.save(toAccount)         [same TX]
       -> transactionRepository.save(withdrawal)    [same TX]
       -> transactionRepository.save(deposit)       [same TX]
       -> fooService.doWriteWithDatabase()          [same TX — Propagation.REQUIRED joins]

   All 6 operations share the SAME transaction. If any fails, ALL are rolled back.


   Other propagation types:
     REQUIRED        => join existing TX or create new (default)
     REQUIRES_NEW    => always create a NEW transaction (suspend existing)
     MANDATORY       => must run inside an existing TX (throws exception if none exists)
     SUPPORTS        => run inside TX if one exists, otherwise run without TX
     NOT_SUPPORTED   => always run without TX (suspend existing if any)
     NEVER           => must NOT run inside a TX (throws exception if one exists)
     NESTED          => run within a nested transaction (savepoint)



ACID Properties
================

   Atomicity     => entire transfer is one unit of work — all or nothing
                    if debit succeeds but credit fails, everything is rolled back

   Consistency   => system goes from one valid state to another
                    balance checks ensure no negative balances
                    both accounts updated correctly or not at all

   Isolation     => concurrent transactions don't see each other's intermediate states
                    READ_COMMITTED prevents dirty reads
                    other transactions see only committed data

   Durability    => once committed, changes survive system failures
                    the updated balances and transaction records persist in PostgreSQL



Hibernate Configuration (application.yml)
===========================================

   spring:
     jpa:
       hibernate:
         ddl-auto: update
       show-sql: true
       properties:
         hibernate:
           format_sql: true

   ddl-auto options:
     none       => don't touch the schema
     validate   => validate schema matches entities (fail if mismatch)
     update     => auto-update schema to match entities (add columns/tables, never drops)
     create     => drop and recreate schema on startup
     create-drop => create on startup, drop on shutdown

   show-sql: true     => print generated SQL to console
   format_sql: true   => pretty-print the SQL for readability



Auto-Configuration Registration
=================================
