.properties vs .yml

Two formats for externalizing configuration

application.properties

Flat key=value format

# Database configuration spring.datasource.url=jdbc:postgresql://localhost:5432/mydatabase spring.datasource.username=postgres spring.datasource.password=mysecretpassword # Transfer settings transfer.limit=10000 # App settings debug=false npci.location=chennai

application.yml

Hierarchical YAML format

debug: true spring: application: name: money-transfer-system-v7 datasource: url: jdbc:postgresql://... username: postgres password: mysecretpassword transfer: limit: 10000
Note: If both files exist and define the same key, application.properties wins (higher priority).

@Value — Injecting Properties

Reading externalized configuration into Spring beans

@Service public class TransferServiceImpl implements TransferService { @Value("${transfer.limit:10000}") private double transferLimit; public void transfer(String from, String to, double amount) { if (amount > transferLimit) { throw new RuntimeException("Exceeds limit"); } // ... perform transfer } }

Syntax

${property.name:defaultValue}

The colon separates the property key from its default. If no source provides the value, the default is used.

Resolution Order

Spring checks each source in priority order:

CLI args → System props → Env vars → .properties → .yml → default

Relaxed Binding

Spring Boot automatically maps between different naming conventions

All of these map to the same value!

Properties / YAML key

transfer.limit

Kebab-case (recommended)

transfer.limit

Environment variable

TRANSFER_LIMIT

System property

transfer.limit

Spring Boot handles the conversion automatically.

No extra configuration needed — just use any naming style that fits your source, and Spring Boot will find the right value.

Custom Auto-Configuration

Conditionally creating beans based on property values

NpciChnAutoConfiguration.java

@Configuration @ConditionalOnProperty( name = "npci.location", havingValue = "chennai" ) public class NpciChnAutoConfiguration { @Bean public String bean1() { return "Bean 1"; } }

How it works

@ConditionalOnProperty

This entire @Configuration class (and all its @Bean methods) only activates when the specified condition is met.

application.properties
npci.location=chennai

Since npci.location=chennai, the condition is TRUE and bean1 is created.

If npci.location were "mumbai" or missing, this configuration class would be completely skipped.

@ConditionalOn* Family

Spring Boot's conditional bean creation annotations

@ConditionalOnProperty

When a property has a specific value

@ConditionalOnProperty(name="npci.location", havingValue="chennai")

@ConditionalOnClass

When a class is on the classpath

@ConditionalOnClass(DataSource.class)

@ConditionalOnBean

When another bean exists in the context

@ConditionalOnBean(DataSource.class)

@ConditionalOnMissingBean

When a bean does NOT exist (provide default)

@ConditionalOnMissingBean(TransferService.class)

@ConditionalOnWebApplication

Only when running as a web application

@ConditionalOnWebApplication

@ConditionalOnExpression

When a SpEL expression evaluates to true

@ConditionalOnExpression("${feature.enabled:false}")
This is how Spring Boot auto-configuration works internally! Each starter uses @ConditionalOn* to decide what beans to create.

Lombok — Less Boilerplate

Annotations that generate code at compile time

Without Lombok (~40 lines)

public class Account { private Long id; private String name; private double balance; public Account() {} public Account(Long id, String name, double balance) { this.id = id; this.name = name; this.balance = balance; } public Long getId() { return id; } public void setId(Long id) { this.id = id; } public String getName() { return name; } public void setName(String name) { this.name = name; } public double getBalance() { return balance; } public void setBalance(double bal) { this.balance = bal; } }

With Lombok (~10 lines)

@NoArgsConstructor @AllArgsConstructor @Setter @Getter public class Account { private Long id; private String name; private double balance; }

4 annotations replace ~30 lines of boilerplate code. Lombok generates getters, setters, and constructors at compile time.
