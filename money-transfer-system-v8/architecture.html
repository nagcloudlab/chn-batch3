JPA Entity Annotations

Mapping Java classes to database tables

Account Entity

@NoArgsConstructor @AllArgsConstructor @Setter @Getter @Entity @Table(name = "accounts", schema = "public") public class Account { @Id @Column(name = "number", nullable = false, unique = true) private String accountNumber; @Column(name = "holder_name") private String accountHolderName; private double balance; }

Annotation Reference

Marks this class as a JPA entity — mapped to a DB table

@Table(name, schema)

Specifies table name and schema

@Id

Primary key field — every entity must have one

@Column(name, nullable, unique)

Maps field to specific column with constraints

Lombok

Transaction Entity — New in v8

Tracking every debit and credit with entity relationships

@Entity @Table(name = "transactions", schema = "public") public class Transaction { @Id @GeneratedValue(strategy = GenerationType.IDENTITY) private Long id; private Double amount; @Temporal(TemporalType.TIMESTAMP) private LocalDateTime timestamp; @Enumerated(EnumType.STRING) private TransactionType type; @ManyToOne @JoinColumn(name = "account_number", referencedColumnName = "number") private Account account; }
@GeneratedValue(IDENTITY)

Database auto-increments the ID. No need to set it manually.

@Enumerated(EnumType.STRING)

Stores enum as string: "DEPOSIT" or "WITHDRAWAL". Avoids fragile ordinals.

@ManyToOne + @JoinColumn

Many transactions belong to one account. Foreign key: transactions.account_number → accounts.number

@Temporal(TIMESTAMP)

Stores both date and time. Options: DATE, TIME, TIMESTAMP.

public enum TransactionType { DEPOSIT, WITHDRAWAL }

@Query — Custom JPQL Queries

Object-oriented queries on entities, not SQL on tables

public interface TransactionRepository extends JpaRepository<Transaction, Long> { @Query("SELECT t FROM Transaction t WHERE t.account.accountNumber = :accountNumber ORDER BY t.timestamp DESC") List<Transaction> findByAccountNumber(String accountNumber); }

JPQL vs SQL

Transaction t — refers to the entity class, not the table name.

Relationship Navigation

t.account.accountNumber — navigates @ManyToOne to Account's field.

Named Parameters

:accountNumber — bound from the method argument automatically.

Still zero implementation code! Spring Data JPA generates the query execution, parameter binding, and result mapping.

@Transactional — Advanced Attributes

Fine-grained transaction control on the transfer method

@Transactional( transactionManager = "transactionManager", isolation = Isolation.READ_COMMITTED, rollbackFor = RuntimeException.class, noRollbackFor = AccountNotFoundException.class, timeout = 30, readOnly = false ) public void transfer(double amount, String from, String to) { ... }
isolation

READ_COMMITTED — prevents dirty reads. Other transactions see only committed data.

rollbackFor

RuntimeException — any runtime exception triggers a full rollback.

noRollbackFor

AccountNotFoundException — don't rollback if account not found.

timeout

30 seconds max. Auto-rollback if exceeded.

readOnly

false — this transaction writes data. Set true for read-only optimization.

transactionManager

Which TX manager bean to use. Default: "transactionManager".

Transaction Propagation

How transactions behave when one @Transactional method calls another

Propagation.REQUIRED (default)

@Service public class FooService { @Transactional(propagation = Propagation.REQUIRED) public void doWriteWithDatabase() { // joins the SAME transaction // if this fails, everything // rolls back } }

Execution Flow (single TX)

transfer() — TX starts here
  accountRepo.save(from) ← same TX
  accountRepo.save(to) ← same TX
  txRepo.save(withdrawal) ← same TX
  txRepo.save(deposit) ← same TX
  fooService.doWrite() ← same TX (REQUIRED joins)
All 6 operations share ONE transaction. If any fails, ALL are rolled back — this is Atomicity.

Propagation Types

Seven propagation strategies for nested transactional methods

Propagation Behavior Use Case
REQUIRED Join existing TX, or create new Default — most common
REQUIRES_NEW Always create a NEW TX (suspend existing) Audit logs that must persist
MANDATORY Must run inside existing TX (throws if none) Enforce caller has a TX
SUPPORTS Run inside TX if one exists, otherwise without Read operations
NOT_SUPPORTED Always run without TX (suspend existing) Long-running reads
NEVER Must NOT run inside TX (throws if one exists) Ensure no TX context
NESTED Run in a nested TX (savepoint) Partial rollback

ACID Properties

How our transfer method guarantees data integrity

Atomicity

The entire transfer is one unit of work. Debit + credit + transaction records — all or nothing. If credit fails after debit, everything rolls back.

Consistency

System goes from one valid state to another. Balance checks prevent negative amounts. Both accounts updated correctly or not at all.

Isolation

READ_COMMITTED — other transactions don't see intermediate states. No dirty reads. Only committed data is visible.

Durability

Once committed, changes survive system failures. Updated balances and transaction records persist permanently in PostgreSQL.

Spring's @Transactional + JPA + PostgreSQL together enforce all four ACID properties on every transfer operation.

Hibernate Configuration

JPA/Hibernate settings in application.yml

application.yml

spring: datasource: url: jdbc:postgresql://... username: postgres password: mysecretpassword jpa: hibernate: ddl-auto: update show-sql: true properties: hibernate: format_sql: true

ddl-auto options

update

Auto-update schema to match entities. Adds columns/tables, never drops. Used in v8.

validate

Validate schema matches entities. Fail on mismatch. Good for production.

create

Drop and recreate schema on every startup.

create-drop

Create on startup, drop on shutdown. For tests.

none

Don't touch the schema at all.
