The Problem in Code
Dependency created inside the method using "new"
public void transfer(double amount, String from, String to) {

  // Problem: creating dependency inside the method   JdbcAccountRepository repo = new JdbcAccountRepository();
  Account fromAcc = repo.loadAccount(from);
  Account toAcc = repo.loadAccount(to);
  // ... transfer logic ...
}

Every call to transfer() creates a brand new JdbcAccountRepository object

Problems
Two Problems
Design issue + Performance issue caused by using "new" inside the method

1. Design Issue — Tight Coupling

// Service is locked to JDBC impl
JdbcAccountRepository repo = new JdbcAccountRepository();

Can't swap — Want JPA? Change service code.
Can't mock — No way to inject a test double.
Can't extend — Adding MongoDB means editing existing code.

2. Performance Issue — Wasteful

// Called 1000 times = 1000 objects
public void transfer(...) {
  new JdbcAccountRepository(); // every time!
}

New object per call — wasteful memory allocation.
New connection per call — slow and expensive.
No reuse — previous object is garbage collected.

Analysis
Root Cause + Impact
Why this happens and what it leads to

Root Cause

The dependent (TransferServiceImpl) creates its own dependency using new
It controls WHAT to use and WHEN to create — no external flexibility

Locked to JDBC

Can't switch to JPA, MongoDB, or any other implementation without changing service code

Hard to Test

Can't mock the repository — unit testing the service layer is nearly impossible

Slow & Wasteful

New object created on every method call — wastes memory, connections, and time

Hard to Maintain

Changing the repository means changing the service — ripple effect across layers
