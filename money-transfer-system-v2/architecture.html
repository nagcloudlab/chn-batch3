Code
Code Comparison — TransferServiceImpl

v1 — Tight Coupling

public void transfer(...) {
  JdbcAccountRepository repo =     new JdbcAccountRepository();
  Account from = repo.loadAccount(...);
  // transfer logic...
}

✗ Creates dependency inside method
✗ Concrete type hardcoded
✗ New object every call

v2 — Constructor DI

private final AccountRepository repo;
private final Notification notif;

public TransferServiceImpl(   AccountRepository repo,   Notification notif) {   this.repo = repo;   this.notif = notif; }
public void transfer(...) {
  Account from = repo.loadAccount(...);
  // transfer logic...
}

✓ Dependencies injected from outside
✓ Interface type — swappable
✓ Single shared instance

Dependency Injection
Constructor vs Setter Injection
Required deps via constructor, optional deps via setter

Constructor Injection (Required)

public class TransferServiceImpl {

  private final AccountRepository repo;
  private final Notification notif;

  public TransferServiceImpl(     AccountRepository repo,     Notification notif) {     this.repo = repo;     this.notif = notif;   }
}

✓ Immutable (final fields)   ✓ Required at creation   ✓ Object always fully initialized

Setter Injection (Optional)

public class TransferServiceImpl {

  private Notification notif;

  public void setNotification(     Notification notif) {     this.notif = notif;   }

  // null-safe usage
  if (notif != null) {
    notif.send(...);
  }

✓ Optional dependency   ✓ Can be absent   ✓ Set after construction

What to do?

▣

Encapsulation

How to do?

Inheritance

Design Principles
SOLID Principles
v2 actively applies O, L, and D
S
Single Responsibility
A class should have only
one reason to change
O
Open-Closed
Open for extension, closed for modification

v2: Add JpaAccountRepo without
changing TransferServiceImpl
L
Liskov Substitution
Subtypes must be substitutable
for their base types

v2: JdbcRepo and JpaRepo
are interchangeable
I
Interface Segregation
Clients should not depend on
interfaces they don't use
D
Dependency Inversion
High-level modules should depend
on abstractions, not low-level modules
