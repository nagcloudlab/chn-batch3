REST — Representational State Transfer

An architectural style, not a protocol — Roy Fielding, 2000

RE

Resources are represented
(usually as JSON)

S

State

Each request contains
all needed state

T

Transfer

State is transferred
between client & server

Everything is a RESOURCE (noun), and we perform
ACTIONS (verbs / HTTP methods) on those resources

Resource: /accounts
+
Actions: GET, POST, PUT, DELETE

REST Rules

6 REST Constraints

What makes an API "RESTful"?

1. Client-Server

Separation of concerns. Client handles UI, Server handles data.

2. Stateless ⭐

Most important for API design! Server stores NO client context. Each request has ALL info needed.

3. Cacheable

Responses can be cached for performance.

4. Uniform Interface

Consistent way to interact with resources.

5. Layered System

Client doesn't know if talking to server or proxy.

6. Code on Demand

Optional. Server can send executable code.

Key Concept

Stateless — Deep Dive

✗ Stateful (Bad for REST)

Request 1: "I am user Raj"
Server: "OK, I'll remember you"

Request 2: "Show my balance"
Server: "Raj's balance is 50,000" ← remembered

What if Request 2 goes to a different server? That server doesn't know who Raj is!

✓ Stateless (REST Way)

Request 1: "I am Raj" + Token: xyz123
Server: "OK, here's your token"

Request 2: "Show balance" + Token: xyz123
Server: Validates token → "Balance: 50,000"

Every request is self-contained. Any server can handle it.

Design Implication: Every request must include auth token! Any server instance can handle any request! Horizontally scalable!

Practical

HTTP Methods — Banking Examples

GET /accounts → List all accounts

GET /accounts/12345 → Get specific account

POST /accounts → Create new account
Body: {"name": "Raj Kumar", "type": "savings"}

PATCH /accounts/12345 → Update only account type
Body: {"type": "current"}

DELETE /accounts/12345 → Close / remove account

⚠️ Key Insight: Same URL + different method = different operation!

Critical Concept

Idempotency — Why It Matters

Same request multiple times = Same result

Idempotent (Safe to retry)

GET /accounts/123

Call 1x → Balance: 50,000
Call 5x → Balance: 50,000
Same result every time ✓

NOT Idempotent (Dangerous!)

POST /transfers {amount: 1000}

Call 1x → Transfer #1 created 💰
Call 2x → Transfer #2 created 💰💰
Duplicate! Money debited twice!

Design Consideration: What happens on network retry? POST might create duplicates (needs idempotency key). GET/PUT/DELETE are safe to retry.

Common Confusion

401 vs 403 — The Bouncer Analogy

401 Unauthorized

"Who are you?"

You → 🚪 → "Show ID please"
"I don't have one" / "ID is fake"

Result: Can't let you in

Fix: Provide valid token / credentials

403 Forbidden

"I know you, but NO!"

You → 🚪 → "Show ID please"
"Here's my valid ID"
"You're Raj, but this is VIP area"

Result: Access denied

Fix: Need higher privileges / different role

No token → 401 • Expired token → 401 • Valid token, wrong role → 403

Gotchas

JSON Rules & Common Mistakes

✓ Valid✗ InvalidWhy
"name": "Raj"'name': 'Raj'Must use double quotes
"amount": 5000"amount": 5,000No formatting in numbers
"active": true"active": TrueBooleans are lowercase

Spring Boot REST

@RestController — Building REST APIs

Spring Boot makes building REST APIs simple

@RestController

@RestController
@RequestMapping("/api/v1/accounts")
public class AccountController {

  @GetMapping
  public List<Account> getAll() {
    return service.findAll();
  }

  @GetMapping("/{id}")
  public Account getById(
    @PathVariable String id) {
    return service.findById(id);
  }
}

@RestController = @Controller + @ResponseBody
Returns JSON directly, not view names

@RequestMapping on class = base URL prefix for all methods

@GetMapping / @PostMapping / @PutMapping / @DeleteMapping

@PathVariable — extracts {id} from URL path

Spring auto-converts Java objects to JSON using Jackson (included in starter-web)

Spring Boot REST

@RequestBody & ResponseEntity

POST — Create Resource

@PostMapping
public ResponseEntity<Account> create(
    @Valid @RequestBody AccountDto dto) {

  Account saved = service.create(dto);

  return ResponseEntity
    .status(HttpStatus.CREATED) // 201
    .body(saved);
}

PUT — Update Resource

@PutMapping("/{id}")
public ResponseEntity<Account> update(
    @PathVariable String id,
    @Valid @RequestBody AccountDto dto) {

  Account updated = service.update(id, dto);

  return ResponseEntity.ok(updated);
}

@RequestBody
Deserializes JSON body → Java object

ResponseEntity
Full control over status code + headers + body

@Valid
Triggers Bean Validation on the DTO

Spring Boot REST

DELETE, @RequestParam & @ResponseStatus

DELETE — Remove Resource

@DeleteMapping("/{id}")
@ResponseStatus(HttpStatus.NO_CONTENT)
public void delete(
    @PathVariable String id) {
  service.delete(id);
  // returns 204 No Content
}

@ResponseStatus — sets the HTTP status code when no ResponseEntity is used

GET with Query Params

// GET /api/v1/accounts?status=active&limit=10

@GetMapping
public List<Account> search(
  @RequestParam String status,
  @RequestParam(
    defaultValue = "20") int limit) {
  return service.search(status, limit);
}

@RequestParam — extracts ?key=value from query string. Supports defaultValue.

Spring Boot REST

REST Exception Handling

Return proper status codes and JSON error responses

@RestControllerAdvice
public class ApiExceptionHandler {

  @ExceptionHandler(AccountNotFoundException.class)
  @ResponseStatus(HttpStatus.NOT_FOUND)
  public Map<String,String> handle(
      AccountNotFoundException ex) {
    return Map.of(
      "error", ex.getMessage(),
      "status", "404");
  }
}

@RestControllerAdvice
= @ControllerAdvice + @ResponseBody
Returns JSON error responses (not views)

@ResponseStatus(NOT_FOUND)
Sets HTTP 404 on the response

Return Map/DTO
Auto-serialized to JSON error body

@ControllerAdvice → returns views (HTML)
@RestControllerAdvice → returns data (JSON)

Reference

REST API Cheat Sheet

Methods: GET=Read • POST=Create • PUT=Replace • PATCH=Update • DELETE=Remove

Status: 2xx=Success • 4xx=Client Error • 5xx=Server Error

Headers: Content-Type: application/json • Authorization: Bearer <token>

JSON: Double quotes only • No trailing commas • Booleans/null lowercase

Stateless: Every request needs full context • Horizontally scalable

Auth: 401 = "Who are you?" • 403 = "No permission" • POST is NOT idempotent

Spring Boot: @RestController • @RequestBody • @PathVariable • ResponseEntity • @RestControllerAdvice

Wrap Up

Key Takeaways

✓ API = Contract between systems (how to request, what you get back)

✓ REST = Architectural style using HTTP — Resources (nouns) + Methods (verbs)

✓ HTTP Methods (CRUD) — GET, POST, PUT, PATCH, DELETE

✓ Status Codes tell the story — 2xx happy, 4xx your fault, 5xx server fault

✓ JSON is the universal data format for APIs

✓ Authentication (who) vs Authorization (what) — JWT is most common

✓ Spring Boot makes building REST APIs simple — @RestController + ResponseEntity
